# GSL-TW V111: อัลกอริทึมแก้ปัญหา VRPTW แบบ Deterministic ประสิทธิภาพสูง

[Read in English](README.md)

**GSL-TW V111** คือเอนจินประมวลผลที่ถูกพัฒนาขึ้นมาโดยเฉพาะเพื่อแก้ปัญหาการจัดเส้นทางเดินรถที่มีกรอบเวลา (Vehicle Routing Problem with Time Windows - VRPTW) โดยมุ่งเน้นไปที่ประสิทธิภาพการคำนวณขั้นสูงสุดและความน่าเชื่อถือแบบกำหนดได้ (Deterministic)

แตกต่างจาก Metaheuristic แบบสุ่ม (Stochastic) ทั่วไป GSL-TW ใช้โครงสร้างเชิงสถาปัตยกรรมขั้นสูงเพื่อระบุคำตอบที่ใกล้เคียงจุดที่ดีที่สุด (Best Known Solutions - BKS) ด้วยความหน่วงเวลาที่ต่ำมาก (Ultra-low latency) ทำให้มันพร้อมสำหรับการนำไปใช้งานจริง (Production-Ready) ในอุตสาหกรรมโลจิสติกส์ขนาดใหญ่

## คุณลักษณะเด่น (Key Characteristics)
* **การรองรับสเกลที่ครอบคลุม (Unified Scalability):** สถาปัตยกรรมแบบ Single-script ที่สามารถแก้ปัญหาได้ทุกระดับ ตั้งแต่ 100 ถึง 1,000 โหนดลูกค้า
* **ประสิทธิภาพบนอุปกรณ์เคลื่อนที่ (Mobile-Optimized Efficiency):** พัฒนาและทดสอบการทำงานทั้งหมดบนสภาพแวดล้อม Mobile Edge (แอปพลิเคชัน Pydroid 3) ซึ่งพิสูจน์ให้เห็นถึงประสิทธิภาพทางโครงสร้างของอัลกอริทึมที่เหนือกว่า โดยไม่ต้องพึ่งพาฮาร์ดแวร์ประมวลผลขนาดใหญ่
* **ความน่าเชื่อถือแบบคงที่ (Deterministic Reliability):** ไม่มีความแปรปรวนจากการสุ่ม (Zero stochastic variance) การประมวลผลด้วยชุดข้อมูลเดิมจะให้ผลลัพธ์ที่คงเดิมและสามารถตรวจสอบทางสถิติได้เสมอ
* **ความเร็วระดับปฏิบัติการจริง (Real-World Speed):** โจทย์ขนาด 1,000 โหนด สามารถประมวลผลสำเร็จภายในเวลาประมาณ 38 วินาที ซึ่งเป็นความเร็วที่เหมาะสมสำหรับการวางแผนปฏิบัติการแบบไดนามิกและเรียลไทม์

## 1. ผลการทดสอบเทียบมาตรฐาน (BKS Match Rate)
เอนจินนี้ได้รับการตรวจสอบอย่างเข้มงวดกับชุดข้อมูลมาตรฐานระดับโลกของ Solomon และ Homberger ครบทุกสเกล (100 ถึง 1,000 โหนด) คำตอบทั้งหมดผ่านการตรวจสอบความถูกต้อง 100% (เป็นไปตามข้อจำกัดด้านความจุและกรอบเวลาอย่างสมบูรณ์)

| ชุดข้อมูล | ขนาดของปัญหา | อัตราการคำนวณจำนวนรถเทียบเท่า BKS | เวลาประมวลผลเฉลี่ย (Mobile Edge) |
| :--- | :--- | :--- | :--- |
| **Solomon-100** | ขนาดเล็ก (100 โหนด) | 46.4% | ~2.78 วินาที |
| **Solomon-200** | ขนาดกลาง (200 โหนด) | 51.7% | ~7.57 วินาที |
| **Homberger-400** | ขนาดใหญ่ (400 โหนด) | 35.0% | ~20.0 วินาที |
| **Homberger-600** | ขนาดใหญ่ (600 โหนด) | 30.0% | ~29.20 วินาที |
| **Homberger-800** | ขนาดใหญ่มาก (800 โหนด) | 35.0% | ~31.15 วินาที |
| **Homberger-1000** | ขนาดใหญ่มาก (1,000 โหนด) | 21.6% | ~38.22 วินาที |

## 2. การเปรียบเทียบประสิทธิภาพกับอัลกอริทึมพื้นฐาน
เพื่อแสดงให้เห็นถึงความสามารถในการทำงานแบบเรียลไทม์ GSL-TW V111 ได้ถูกนำไปทดสอบเปรียบเทียบกับอัลกอริทึมที่มีชื่อเสียง ภายใต้ **ข้อจำกัดของเวลาการประมวลผลที่เท่าเทียมกัน** บนอุปกรณ์เคลื่อนที่เครื่องเดียวกัน

* **GSL เทียบกับ Solomon I1 (Insertion Heuristic):**
  * **100 โหนด:** GSL ชนะ 45/56. (เวลาเฉลี่ย: GSL 2.77s vs I1 1.49s)
  * **200 โหนด:** GSL ชนะ 53/60. (เวลาเฉลี่ย: GSL 7.43s vs I1 9.83s)
  * **800 โหนด:** GSL ชนะ 57/60. (เวลาเฉลี่ย: GSL 30.57s vs I1 181.06s)
  * *ข้อค้นพบ:* อัลกอริทึมพื้นฐาน (Constructive Heuristic) มีปัญหาด้านการขยายสเกลอย่างรุนแรงเมื่อข้อมูลมีขนาดใหญ่ ในขณะที่เอนจินแบบ Deterministic (GSL) ยังคงรักษาเวลาการประมวลผลเชิงเส้นที่เสถียรไว้ได้อย่างสมบูรณ์แบบ

* **GSL เทียบกับ ALNS (Adaptive Large Neighborhood Search):**
  * **100 โหนด:** GSL ชนะ 53/56.
  * **800 โหนด:** GSL ชนะ 60/60. (เวลาเฉลี่ย: GSL 30.57s vs ALNS 380.43s)
  * *ข้อค้นพบ:* ภายใต้ข้อจำกัดของเวลาปฏิบัติการจริง กลไกการทำลายและสร้างใหม่แบบสุ่มของ ALNS ไม่สามารถหาคำตอบที่เหมาะสมได้ GSL ประมวลผลได้เร็วกว่ากว่า 12 เท่า พร้อมมอบประสิทธิภาพการจัดเส้นทางที่เหนือกว่า Metaheuristic อย่างไร้ข้อกังขา

* **GSL เทียบกับ Tabu Search (TS):**
  * **100 โหนด:** GSL ชนะ 54/56. (เวลาเฉลี่ย: GSL 2.77s vs TS 139.90s)
  * *ข้อค้นพบ:* อัลกอริทึมสาย Trajectory-based เช่น TS มีภาระการคำนวณที่หนักเกินไป ($O(N^3)$) สำหรับการใช้งานบน Edge Device โดยทำงานช้ากว่า GSL ถึง 50 เท่า

## กลยุทธ์ทางเทคนิคและสถาปัตยกรรม (Technical Strategy)
เอนจิน GSL แทนที่การใช้พลังประมวลผลมหาศาลแบบสุ่ม ด้วยกลยุทธ์หลัก 3 ประการ:
1. **Geometric Sensing:** การระบุคลัสเตอร์โครงสร้างเริ่มต้นที่มีคุณภาพสูง โดยอาศัยกฎเกณฑ์เชิงพื้นที่
2. **Deterministic Vehicle Minimization:** การให้ความสำคัญกับการหั่นลดจำนวนยานพาหนะให้ได้ระดับ BKS ก่อนเป็นอันดับแรก 
3. **Temporal Constraint Synchronization:** การจัดการกรอบเวลา (Time Windows) อย่างแม่นยำ โดยไม่ทำให้ความซับซ้อนของคณิตศาสตร์เพิ่มขึ้นทวีคูณ

## บริการและความร่วมมือทางธุรกิจ

เปิดรับการร่วมงานและให้คำปรึกษาระดับมืออาชีพ ในขอบเขตดังต่อไปนี้:

- **Logistics-as-a-Service (LaaS):** บริการระบบประมวลผลเส้นทางแบบเรียลไทม์สำหรับฟลีทรถขนส่งระดับองค์กร
- **High-Precision Modeling:** การออกแบบและพัฒนาอัลกอริทึมเฉพาะทาง สำหรับแก้ปัญหาซัพพลายเชนที่มีเงื่อนไข (Constraints) ซับซ้อนขั้นสูง
- **Technical Consultancy:** บริการให้คำปรึกษาทางเทคนิค, ทดสอบขีดจำกัดระบบเครือข่ายขนาดใหญ่ (Stress-testing) และการตรวจสอบประเมินประสิทธิภาพการจัดเส้นทาง (Optimization audits)
